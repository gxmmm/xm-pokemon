# PROJECT_RULES.md — AI 开发最高规则

> 所有 AI Agent 在动手前**必须首先阅读**本文档与 `README.md`。

## 一、项目定位
Pokemon Online 是一款**朋友共同探索、共同收集、共同培养**的在线宝可梦游戏。
**不是**商业 MMO、排位竞技、数值内卷或放置挂机游戏。任何新增系统都必须符合这一定位。

## 二、已冻结（Frozen）设计 — 不得修改
1. 朋友共同探索，不做商业 MMO。
2. PVE 与 PVP 均为同时上场（3vN）；野外遇敌一次出现 1~3 只宝可梦。
3. 随身携带最多 20 只，暂无仓库（未来保留）；PVE/PVP 各一套 3 只阵容 + 自由摆位阵型（起始阵位）。
4. 战斗结束自动回满状态（全队 HP 与异常状态清空）。
5. 捕获 100% 成功；携带满 20 只时无法捕捉，需先放生。
6. 无排行榜、无世界 Boss、无每日签到、无体力、无 VIP。
7. 炼妖随机继承主宠或副宠种族，**不产生融合种族**。
8. 主动技能独立 CD，无 MP/PP；保留普通攻击保证持续输出。
9. 性格决定 **AI 行为**（不是简单属性加减）。
10. 前端负责计算，后端负责存档。
11. 关系型数据库（Cloudflare D1）。
12. 静态配置与动态存档彻底分离（Species vs PokemonInstance）。
13. **资质(IV)按稀有度设上限**：common24 / uncommon26 / rare29 / legendary·mythical31，野生与捕获在此上限内随机（最低1）。**炼妖资质不封顶**：后代每项资质 = (主+副)/2 × (0.8~1.2)，最低1、无上限（可超31/超种族上限，详情标"超限"）。**成长(growth)封顶1.3**、按稀有度 common1.1 / uncommon1.15 / rare1.2 / legendary·mythical1.3；炼妖时成长按父母均值±0.06波动，封顶种族上限、**不可突破1.3**。详情/图鉴须展示资质与成长上限。
14. **梦幻被动池（每族固定）**：每个宝可梦模型有独立固定的4~5个被动池（按 id 确定性生成，同类型也不同）；每族1~2个**必带被动**（天生持有，不雷同）；野生/捕获随机持有1~5个（含必带，受池大小限制）；炼妖被动 = 后代模型必带100%保留 + 父母被动并集逐项55%保留、**封顶24**（`PASSIVE_SKILL_MAX=24`，炼妖可跨代积累广被动池）。详情须展示已拥有+未拥有（置灰"未拥有"）。
15. **地图遇敌**：草丛(tile 3)遇敌不变；洞穴、水域、深空遗址等自然地面地图设 `encounterFloor`，自然地面每步可遇敌（这些图无草丛瓦片）。各地图遇敌表按难度分级稀有度（后期图出稀有/传说/幻兽）。
16. **开发训练塔**：雾湾镇的「幻境之塔」是战斗/捕捉/炼妖测试沙盒，可提供超出主线节奏的分层等级遭遇；必须由 `ILLUSION_TOWER_ENABLED` 配置开关统一控制，正式剧情版本可隐藏入口、地图与世界地图节点，不影响存档。
17. **Pixi 是唯一正式战斗 renderer**：世界与战斗统一使用 Pixi GPU renderer。`BattleSim → presentation → renderer DTO` 的权威分层不得改变；正式战斗不得接入 Three.js、GLB 或玩家 renderer 切换。
18. **Pixi 序列帧战斗美术**：战斗角色使用经 manifest 声明的 2D 静态图或序列帧资产。动作、挂点、调色主题、技能/被动/特性/状态特效、环境反应、资源 manifest 与品质预算均须由静态配置和通用 resolver 驱动；禁止 renderer/Vue 按 `speciesId`/`skillId`/资源路径硬编码。


## 三、协作语言（全局记忆）
- 与开发者的所有沟通、进度说明、问题说明、交付总结和计划列表，统一使用**简体中文**。
- 代码标识符、命令、文件路径、第三方 API 名称及必须保留的技术术语可维持原始语言；其余说明应使用简体中文。
- 新建或更新面向协作的计划、交接和任务文档时，默认使用简体中文；除非开发者明确要求其他语言。

## 四、八条最高设计原则
1. 探索优先（鼓励主动探索，而非重复刷同一地点）
2. 收集优先（获得容易，培养困难）
3. 培养优先（长期目标永远是培养/炼妖/优化/展示）
4. 朋友优先（促进合作分享，不制造恶性竞争）
5. 世界优先（宝可梦/NPC/Boss 生活在地图里，不是菜单）
6. AI 优先（拼培养理解阵容，不拼操作；实时战斗全由 AI 完成）
7. 配置优先（技能/地图/宝可梦/特性/NPC，以及模型/动作/特效/资源映射/品质预算全部配置化，避免硬编码）
8. 扩展优先（新增世代/地图/技能只增配置，不推翻架构）

## 五、明确禁止
排行榜、世界 Boss 抢夺、每日签到、强制每日任务、体力限制、VIP、战力排行、强制在线奖励、世界频道刷屏、自动挂机刷怪、高失败率捕获、重复刷材料作为主要玩法。

## 六、明确鼓励
隐藏地图/隐藏 NPC/隐藏剧情、宝可梦行为、地图生态、收集图鉴、炼妖深度、宝可梦个性化、与朋友分享发现、世界细节。

## 七、技术原则
- 后端只负责数据保存（存档服务器，不是战斗服务器）。
- 能前端计算，就不要后端计算。
- 所有数据分两类：**静态配置**（宝可梦/技能/特性/性格/地图/NPC/模型/动作/特效/资源映射）与**动态数据**（玩家/实例/背包/好友/炼妖记录）。静态配置不得写入玩家存档。
- engine 决定战斗事实；presentation 将结构化事件变为 cue；renderer 只消费 DTO。renderer 不读取 Pinia/BattleSim 内部状态、不重算结果、不改变地图/战斗规则。模型碰撞、单位个人间距、接近停靠与站位冲突属于 engine 权威空间规则，不得只在 renderer 修正。
- 新增 Pixi 战斗资产必须有 schema/reference 校验、asset manifest、动作 clip、质量档预算和资产来源/许可证记录。资源路径仅由 manifest 解析，不能在 renderer 或 Vue 中拼接物种路径。
- `castTime` 是技能玩法权威的前摇。动作的 prepare/windup/charge/release/channel/impact/recovery 仅控制表现时间轴；任何改变可打断、持续结算或伤害时机的需求必须先作为 engine/玩法设计单独批准。

## 八、AI 开发黄金规则
1. 先阅读 `README.md` 与本文档，再阅读对应模块代码。
2. 不要新增商业化玩法。
3. 不要修改已冻结设计。
4. 所有新增功能优先配置化（放 `packages/config`）；战斗美术同时更新 art profile、asset manifest、引用校验与回归基线。
5. 保持低压力、高探索、高收藏价值。
6. 实现时优先保证架构可扩展。
7. 任何新增系统须回答三个问题：是否增加探索乐趣？是否增加培养深度？是否违背朋友共同游玩理念？
8. 新增或修改 Pixi 战斗前必须确认：它是 engine 玩法、presentation choreography，还是仅美术差异？共享技能是否仍只有一份定义？是否引入了按物种/技能 ID 的 renderer 硬编码？资产是否已有 manifest、动作 clip 与来源记录？
